# Elementor Pro + Axxiom AEO

Sites use **Elementor Pro Theme Builder** with a **Post Content** widget on the single-post template. The Axxiom platform publishes **HTML into `post_content`** via REST API; Elementor wraps that body in your theme shell.

**Plugin:** [`axxiom-aeo-schema.php`](axxiom-aeo-schema.php) (v1.1.0+) — registers `aeo_schema_json` for REST and prints JSON-LD in `<head>`.

---

## How publish works with Elementor

```
Axxiom Approve & Publish
    → REST POST /wp/v2/posts { content: HTML, meta: { aeo_schema_json } }
    → post_content (Claude HTML)
    → Elementor Single Post template → Post Content widget renders HTML
    → MU plugin → JSON-LD in <head>
```

No Elementor widget JSON is generated by the app. Use Elementor for **layout chrome** (header, footer, sidebars); use **Post Content** for the article body.

---

## Theme Builder checklist (per site)

1. **Templates → Theme Builder → Single Post**
2. Confirm the template includes **Post Content** or **Theme Post Content** widget (required).
3. Confirm display conditions: **All Posts** (or the categories you use for AEO content).
4. Publish / update the template if you change it.

### After Axxiom publishes a post

- **Do** use **Edit post** (block/classic) for title/slug tweaks if needed.
- **Do not** use **Edit with Elementor** on the post body unless you intend to replace API HTML — saving in the builder can add `_elementor_data` and overwrite `post_content`.

API-created posts stay in **classic content mode** inside an Elementor **template** (correct behavior).

## REST API meta (v1.1.0+)

The MU plugin registers `aeo_schema_json` with `show_in_rest => true` for **posts** and **pages**.

Without this, WordPress may **ignore** schema sent by the backend even when publish returns success.

**Verify after one publish:**

1. wp-admin → open the post → **Custom Fields** (or use a meta inspector plugin).
2. Field `aeo_schema_json` should contain JSON text.
3. View page source → one or more `application/ld+json` blocks from Axxiom (see schema dedup below).

Re-upload [`axxiom-aeo-schema.php`](axxiom-aeo-schema.php) to all sites after upgrading from v1.0.0 — see [WP_ENGINE_SETUP.md](WP_ENGINE_SETUP.md).

## Avoid duplicate JSON-LD (Yoast / Elementor)

Multiple plugins outputting schema causes duplicate or conflicting structured data.

### Yoast SEO (installed on your sites)

Per site, in wp-admin:

1. **SEO → Settings → Site features → Schema** (or **Search Appearance → Schema** depending on Yoast version).
2. For **AEO-published blog posts**, prefer schema from **`aeo_schema_json`** only.
3. Practical approach:
   - Keep Yoast schema for **non-AEO** pages if desired.
   - For posts published from Axxiom, either disable Yoast’s Article/WebPage output for those posts, or accept that Yoast may add extra graph nodes — **test page source** after publish.
4. **Recommended:** After pilot publish, view source and count `application/ld+json` scripts. Aim for **one primary FAQ/Article block** from Axxiom plus no conflicting duplicate `@type` for the same entity.

### Elementor

- Elementor Pro does not replace `wp_head` for singles using Theme Builder + Post Content.
- If you use Elementor add-ons that inject schema, disable them for blog singles or document the overlap.

### Schema dedup audit checklist (per site)

- [ ] Publish one test post from Axxiom Content Review
- [ ] View page source → search `application/ld+json`
- [ ] Confirm `aeo_schema_json` content appears (FAQPage / Article from platform)
- [ ] Note any **additional** JSON-LD from Yoast or others
- [ ] Adjust Yoast schema settings if duplicates conflict
- [ ] Re-run **Schema Health** in Axxiom dashboard on that URL
